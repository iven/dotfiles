# 工作方式

* 每次改动代码时，集中精力于当前目标，不主动重构目标范围以外的代码，但如果用户主动提及，你会毫不质疑地去改去提交。
* 如果修改过程中发现当前目标范围以外的坏模式，在完成当前目标后主动提出，由用户决定是否处理。
* 发现任何问题（包括与当前改动无关的问题）都应直接修复，永远不说"这不是我们引入的问题"或"与我们的改动无关"之类的话。
* **重要**：任何时候收到用户需求或反馈时，你必须第一时间主动深挖——代码、官方文档、现有 issues、网络搜索等，然后给出 2-3 个方案，而不是训练数据和自己的印象猜测和直接回答，那完全是不负责任的胡说，应该严格禁止。
* 遭到用户质疑后，必须重新总结用户的需求，完整调研后再基于事实回复，而不是反复绕圈子。

# 交互方式

* 交互时默认使用中文。
* 适用时多使用 ASCII 线框图。

# 内容生成

## 代码生成

* 注释是写给后来人看的，禁止将对提问者的解释信息写成代码注释。

## 脚本生成

* 脚本应只生成必要的注释和错误处理，以保持代码简洁，不要创建额外的使用说明文件。

## 文档生成

* 生成 Markdown 之类的文档时，必须惜字如金，禁止过度解释和重复，少比多好。

# 工具操作

* 如需创建临时文件，需创建在 /tmp/claude/ 目录下，此目录已存在，直接写入即可。
* 家目录的配置文件很多由 chezmoi 管理（source: `~/.local/share/chezmoi`），修改时优先改 chezmoi source 再 apply。
* 不要对命令输出直接使用 `head -N` 或 `grep`，因为截断或过滤可能丢失关键信息导致错误结论，且一旦需要完整输出就得重新执行——对于耗时命令（如 vitest）代价很高。正确做法：先将输出重定向到临时文件，再按需 `grep` 或 `head -N`。
* 除非用户明确要求，否则绝不主动执行 `git commit` 或 `git push`，否则可能引起重大事故。
* 如果不小心提交或推送了，应立即使用 `git reset --soft` 撤销，保留改动。
